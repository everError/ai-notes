# 하네스 엔지니어링 (Harness Engineering)

> LLM 시대에 성능 차별화의 무게중심이 "프롬프트 문구"에서 "모델을 감싸는 실행 환경 전체의 설계"로 이동한 것을 가리키는 개념.
> 2026년 상반기 에이전트 생태계(Claude Code, Codex CLI, ZCode 등)의 핵심 키워드.

---

## 1. 하네스란 무엇인가

### 어원

- 말의 **마구(馬具, harness)** 에서 유래. 동력원(말)을 실제 일(수레 끌기)에 연결하는 장치.
- 핵심 은유: **동력원 자체는 일을 못 한다. 동력을 일에 연결하는 장치가 필요하다.**

### 소프트웨어 엔지니어링의 원래 용례 — 테스트 하네스

AI 이전부터 존재하던 정식 엔지니어링 용어.

| 구성            | 역할                         |
| --------------- | ---------------------------- |
| 테스트 드라이버 | 테스트 대상 코드를 호출/실행 |
| 스텁 / 목(mock) | 아직 없는 의존성을 대신함    |
| 실행 환경       | 셋업, 티어다운, 격리         |
| 결과 수집기     | 로그, 리포트, 성공/실패 판정 |

→ 공통점: **대상 코드 자체가 아니라, 대상을 "돌리고 관찰하기 위한" 주변 장치 전체**를 하네스라고 부른다.

---

## 2. 에이전트 하네스 (Agent Harness)

### 정의

LLM(모델)을 감싸서 실제 작업을 수행 가능하게 만드는 **실행 루프 + 주변 시스템 전체**.

```
LLM 단독:        텍스트 in → 텍스트 out  (그 이상 아무것도 못 함)
LLM + 하네스:    목표 in → 계획 → 도구 실행 → 관찰 → 반복 → 결과물 out
```

### 구성 요소 (아키텍처)

```
┌─────────────────────────────────────────────┐
│                  에이전트 하네스               │
│                                             │
│  ┌─────────┐   ┌──────────────────────────┐ │
│  │ 시스템    │   │        에이전트 루프        │ │
│  │ 프롬프트  │──▶│  계획 → 도구호출 → 관찰 →   │ │
│  │ / 스킬   │   │  판단 → (반복 or 종료)      │ │
│  └─────────┘   └────────────┬─────────────┘ │
│                             │               │
│  ┌──────────┐  ┌───────────▼────────────┐  │
│  │ 컨텍스트/  │  │        도구 계층          │  │
│  │ 메모리 관리 │  │ 파일IO, bash, 검색, MCP  │  │
│  └──────────┘  └───────────┬────────────┘  │
│                             │               │
│  ┌──────────┐  ┌───────────▼────────────┐  │
│  │ 가드레일   │  │   에러 처리 / 재시도 /    │  │
│  │ 권한/샌드박스│ │   결과 검증(테스트 실행)    │  │
│  └──────────┘  └────────────────────────┘  │
└──────────────────────────────┬──────────────┘
                               │
                          ┌────▼────┐
                          │   LLM   │  ← 교체 가능한 "동력원"
                          └─────────┘
```

| 레이어                   | 하는 일                                           | 예시                                                           |
| ------------------------ | ------------------------------------------------- | -------------------------------------------------------------- |
| **에이전트 루프**        | 계획→실행→관찰→재계획 사이클 제어, 종료 조건 판단 | ReAct 루프, tool-use 루프                                      |
| **도구 계층**            | 모델에게 손발 제공                                | 파일 편집, bash, 웹 검색, MCP 서버                             |
| **컨텍스트/메모리 관리** | 한정된 컨텍스트 창에 뭘 넣고 뺄지 결정            | 세션 압축(claude-mem), 컴팩션, 지식그래프(Understand-Anything) |
| **스킬/지식 주입**       | 모델이 모르는 환경 지식을 파일로 주입             | SKILL.md, CLAUDE.md, 커서룰                                    |
| **가드레일/권한**        | 위험 행동 차단, 샌드박스, human-in-the-loop       | 파일 쓰기 승인, 결제 차단                                      |
| **에러 처리/검증**       | 실패 감지, 재시도, 자기 검증                      | 테스트 자동 실행, 린트, 데모 영상 생성                         |

### 대표 사례

- **Claude Code, Codex CLI, Gemini CLI** = "모델 + 하네스" 패키지 제품
- **ZCode** (Zhipu, 2026.6) = GLM-5.2 전용 하네스. 모델사가 하네스까지 수직 통합한 사례
- **ECC, claude-mem, Understand-Anything** = 하네스의 특정 레이어(메모리, 코드 인덱싱)를 파고드는 오픈소스

## 3. 프롬프트 엔지니어링 vs 하네스 엔지니어링

|                    | 프롬프트 엔지니어링   | 하네스 엔지니어링                            |
| ------------------ | --------------------- | -------------------------------------------- |
| **최적화 대상**    | 단일 요청의 입력 문구 | 실행 루프 전체 시스템                        |
| **단위**           | 문장, few-shot 예시   | 도구 셋, 메모리 정책, 권한 모델, 재시도 로직 |
| **상태**           | 무상태 (1회 호출)     | 유상태 (수십~수백 턴, 세션 간 지속)          |
| **실패 처리**      | 사람이 다시 물어봄    | 하네스가 감지·복구·재시도                    |
| **성능 좌우 요인** | 표현력, 예시 품질     | 컨텍스트 효율, 도구 설계, 검증 루프          |
| **비유**           | 지시서를 잘 쓰는 것   | 작업장 전체를 설계하는 것                    |

핵심 전환: **같은 모델이라도 하네스가 다르면 결과가 크게 달라진다.**
벤치마크에서도 "모델 점수"와 "하네스 성능"을 분리해서 봐야 한다는 관점이 표준화되는 중
(예: 2026.6 semgrep의 GLM-5.2 vs Claude 비교 — 하네스 차이가 결과에 미치는 영향 논의).

## 4. 왜 지금 뜨는가

1. **모델 성능의 수렴** — 프론티어 모델 간 격차가 좁아지면서, 차별화 지점이 모델 밖(하네스)으로 이동.
2. **에이전트의 상용화** — 단발 Q&A가 아니라 수백 턴짜리 장기 작업이 일상화 → 루프/메모리/복구 설계가 곧 제품 품질.
3. **컨텍스트 = 비용** — 토큰 낭비가 직접 비용으로 환산되면서 "컨텍스트에 뭘 넣을지"가 엔지니어링 문제가 됨.
4. **보안 요구** — 프롬프트 인젝션(role confusion 등)에 대한 방어는 모델 훈련만으로 불충분 → 하네스 레벨의 신뢰 경계 설계 필수.
5. **스킬 생태계** — 지식 주입이 파인튜닝이 아닌 마크다운 파일(스킬) 배포로 해결되면서, 하네스가 스킬의 런타임 역할.

---

## 5. 실무 원칙 (체크리스트)

- [ ] **스킬에는 모델이 모르는 것만 넣는다** — 모델이 이미 아는 API 문서를 스킬에 넣으면 토큰 낭비 (MS "Stop overloading your skills")
- [ ] **되돌릴 수 없는 액션은 하네스가 차단** — 결제, 자격증명, 프로덕션 배포는 human-in-the-loop 강제
- [ ] **컨텍스트 예산을 계측** — 세션 압축/컴팩션 도입 전후 토큰 소비 비교
- [ ] **자기 검증 루프를 넣는다** — 에이전트가 테스트/린트/데모 영상으로 자기 작업의 증거를 남기게 설계
- [ ] **모델 교체 가능성 유지** — 하네스와 모델을 결합도 낮게 설계하면 모델 갱신 시 하네스 자산이 그대로 이월됨
- [ ] **평가는 하네스 포함으로** — 모델 단독 벤치마크가 아니라 "우리 하네스 + 후보 모델" 조합으로 측정

---

## 6. 한 줄 요약

> **모델은 엔진, 하네스는 차체.**
> 2026년의 경쟁력은 더 좋은 엔진을 고르는 것보다, 엔진을 갈아 끼워도 굴러가는 좋은 차체를 만드는 데서 나온다.
