# 문서 임베딩 파이프라인 정리 — PDF·Excel을 검색 가능하게 만들기

## 1. 전체 파이프라인

```
문서 (PDF, Excel, ...)
  → ① Parse    텍스트·구조 추출
  → ② Chunk    검색 단위로 분할
  → ③ Embed    벡터로 변환
  → ④ Store    벡터 DB에 저장 (+ 메타데이터)
  → ⑤ Retrieve 질의 벡터와 유사도 검색
```

RAG 품질의 상한은 대부분 **①~②에서 결정**된다. 파싱이 깨진 텍스트는 아무리 좋은 임베딩 모델도 살리지 못한다 ("garbage in, garbage out"). 모델 교체보다 파싱·청킹 개선이 효과가 큰 경우가 많다.

## 2. PDF 파싱

PDF는 "텍스트 문서"가 아니라 **좌표에 글자를 찍는 인쇄 포맷**이다. 읽기 순서, 단락 경계, 표 구조가 파일에 없어서 파서가 추측해야 한다 — 도구마다 품질이 갈리는 이유.

| 도구            | 특징                                             | 적합                        |
| --------------- | ------------------------------------------------ | --------------------------- |
| **PyMuPDF**     | 빠름(C 기반), 텍스트·좌표·이미지 추출            | 기본 선택지                 |
| pypdf           | 순수 파이썬, 가벼움                              | 단순 텍스트, 의존성 최소화  |
| pdfplumber      | 표 추출에 강점 (선 기반 감지)                    | 표 많은 문서                |
| **Docling**     | IBM 오픈소스. 레이아웃 ML 모델로 구조(제목/표/목록) 복원, markdown 출력 | 고품질 필요, 복잡한 문서    |
| Unstructured    | 다양한 포맷 통합, 파티셔닝 API                   | 여러 포맷 일괄 처리         |

핵심 판단 기준:

- **텍스트 PDF vs 스캔 PDF** — 스캔본은 글자가 이미지라서 OCR 필요 (Tesseract, PaddleOCR 등). `page.get_text()`가 빈 문자열이면 스캔본이다
- **표가 중요한가** — 일반 파서는 표를 읽기 순서대로 뭉개버린다. 표가 핵심이면 pdfplumber/Docling으로 구조를 살려 **markdown 표로 변환** 후 임베딩

## 3. Excel / 표 데이터

표 데이터는 텍스트 문서와 근본적으로 다르다: **의미가 셀 값이 아니라 "행 × 열 교차"에 있다.** 시트를 통째로 텍스트화하면 열 헤더와 값의 연결이 끊겨 검색이 망가진다.

전략 (권장 순):

1. **행 단위 직렬화** — 각 행을 "헤더: 값" 문장으로 변환해 행 하나 = 문서 하나

   ```
   제품명: 볼트 M8 | 카테고리: 체결부품 | 재고: 1200 | 단가: 350원
   ```

   행 단위 질의("볼트 M8 재고 얼마?")에 강함. 가장 일반적

2. **markdown 표 변환** — 작은 표를 통째로 하나의 청크로. 표 전체 맥락이 필요한 질의에 유리
3. **요약 + 원본 참조** — 시트 요약문을 임베딩하고, 매칭 시 원본 데이터를 별도 조회 (표가 크고 집계 질의가 많으면 애초에 RAG보다 **Text-to-SQL/DataFrame 에이전트**가 정답인 경우도 많다)

메타데이터에 시트명, 행 번호를 남겨서 검색 결과에서 원본으로 돌아갈 수 있게 한다.

## 4. 청킹(Chunking)

임베딩 모델 입력 한도와 검색 정밀도를 위해 문서를 자르는 단계.

| 전략                | 방식                                        | 비고                             |
| ------------------- | ------------------------------------------- | -------------------------------- |
| Fixed-size          | N자/N토큰 단위 + overlap                    | 단순, 문장이 중간에 잘림         |
| **Recursive**       | 단락→문장→단어 순으로 구분자를 바꿔가며 분할 | 사실상 표준 기본값               |
| Structure-aware     | 제목/섹션 경계 기준 (markdown 헤더 등)      | Docling류 파서와 조합 시 강력    |
| Semantic            | 문장 임베딩 유사도가 꺾이는 지점에서 분할   | 비용 높음, 효과는 문서에 따라    |

파라미터 감각:

- **chunk size**: 작으면 정밀하지만 맥락 부족, 크면 맥락은 있지만 임베딩이 뭉개짐(여러 주제가 한 벡터에). 보통 300~1000자에서 시작
- **overlap**: 경계에서 잘린 문장 보완. size의 10~20%
- 청크에 **문서 제목·섹션 경로를 prepend** ("[제품 매뉴얼 > 3장 설치] ...")하면 짧은 청크의 맥락 손실을 크게 줄인다 (contextual chunking)

## 5. 임베딩 모델 선택

| 모델                        | 차원 | 특징                                        |
| --------------------------- | ---- | ------------------------------------------- |
| multilingual-e5-small/base  | 384/768 | 가볍고 빠름, 한국어 준수. **prefix 필수** |
| **BAAI/bge-m3**             | 1024 | 다국어 최상급, 8192 토큰, dense+sparse 하이브리드 |
| OpenAI text-embedding-3-*   | 1536/3072 | API 호출, 무난한 품질, 로컬 불가       |

- **한국어 문서면 반드시 다국어 모델** — 영어 전용 모델(예: 구형 all-MiniLM)은 한국어 성능이 크게 떨어진다. 벤치마크는 MTEB(다국어 임베딩 리더보드) 참고
- **e5 계열의 prefix 규약** — 학습 시 약속된 접두어를 붙여야 제 성능이 나온다: 문서에는 `passage: `, 질의에는 `query: `. 빼먹으면 조용히 품질만 떨어지는 흔한 실수
- **차원 수 = 저장 비용·검색 속도와 트레이드오프.** 클수록 좋은 게 아니라 코퍼스 규모와 지연 요구에 맞춘다
- 로컬 실행: sentence-transformers(GPU 가속 가능) 또는 Ollama(`bge-m3` 등 제공, API로 호출)

## 6. 효율 (대량 인제스트 최적화)

임베딩 인제스트는 전형적인 배치 워크로드다. 순진하게 짜면 1건씩 처리하게 되는데, 차이가 수십 배 난다.

1. **배치 인코딩** — `model.encode(texts, batch_size=64)`처럼 묶어서. GPU면 배치가 클수록 처리량 급증 (vLLM의 continuous batching과 같은 원리)
2. **콘텐츠 해시로 중복 제거 + 증분 인제스트** — `sha256(청크 텍스트)`를 벡터 DB의 id로 쓰면:
   - 같은 내용 재실행 시 upsert가 덮어써서 중복이 안 쌓임 (멱등성)
   - 문서 재인제스트 때 변경된 청크만 다시 임베딩 (임베딩 비용이 지배적이므로 효과 큼)
3. **정규화(normalize) 후 저장** — 벡터를 단위 길이로 정규화하면 코사인 유사도 = 내적이 되어 검색이 빨라진다. 대부분 모델이 `normalize_embeddings=True` 옵션 제공
4. **메타데이터 필터 우선** — "2024년 매뉴얼에서만 검색" 같은 조건은 벡터 검색 전에 메타데이터로 거른다. 검색 정확도·속도 모두 개선
5. **파싱과 임베딩 분리** — 파싱 결과(청크 + 메타데이터)를 중간 산출물(jsonl 등)로 저장해두면 모델 교체 시 파싱을 반복하지 않는다

## 7. 품질 확인

인제스트가 끝나면 최소한의 스모크 테스트: 대표 질문 몇 개로 검색해서 **기대한 청크가 top-k 안에 오는지** 확인한다. 체계적인 평가(Recall@k, MRR, Ragas의 context precision)는 llmops-notes의 evaluation에서 다룬다 — 평가 데이터가 있어야 청킹 파라미터 튜닝이 감이 아니라 측정이 된다.
