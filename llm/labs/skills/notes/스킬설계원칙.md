# 스킬 설계 원칙 (Skill Design Principles)

> 스킬 = 에이전트 컨텍스트에 **조건부로 주입되는 지식 파일**(주로 마크다운).
> 파인튜닝 없이, 코드 배포 없이, "모델이 특정 상황에서 어떻게 행동해야 하는가"를 파일 하나로 배포하는 단위.
> 하네스 아키텍처에서 **지식 주입 레이어**에 해당한다.

---

## 0. 스킬의 동작 원리 (전제 지식)

### 3단계 로딩 (Progressive Disclosure)

스킬은 전체가 컨텍스트에 상주하지 않는다. 3단계로 나뉘어 필요한 만큼만 로드된다.

| 단계             | 내용                           | 컨텍스트 비용 | 로드 시점               |
| ---------------- | ------------------------------ | ------------- | ----------------------- |
| 1. 메타데이터    | name + description (~100단어)  | 항상 상주     | 세션 시작               |
| 2. SKILL.md 본문 | 지침 (<500줄 권장)             | 트리거 시     | 관련 작업 감지          |
| 3. 번들 리소스   | references/, scripts/, assets/ | 필요 시       | 모델이 판단해 개별 로드 |

### 트리거 메커니즘 — 알아야 할 특성 2가지

1. **모델은 description만 보고 스킬 사용 여부를 결정한다.** 본문이 아무리 좋아도 description이 부실하면 스킬은 존재하지 않는 것과 같다.
2. **모델은 혼자 쉽게 할 수 있는 작업에는 스킬을 참조하지 않는다.** 단순 1스텝 작업("이 PDF 읽어줘")은 description이 완벽해도 트리거되지 않을 수 있다. 스킬은 복잡한·다단계·특화 작업에서 발동한다.

### 디렉토리 해부

```
skill-name/
├── SKILL.md              # 필수: YAML frontmatter(name, description) + 지침
└── (선택)
    ├── scripts/          # 결정적·반복 작업용 실행 코드 (로드 없이 실행 가능)
    ├── references/       # 필요 시에만 컨텍스트에 로드되는 상세 문서
    └── assets/           # 출력물에 쓰이는 파일 (템플릿, 아이콘, 폰트)
```

---

## 원칙 1. 모델이 모르는 것만 담는다 (최우선)

MS "Stop overloading your skills"의 핵심. 모델이 이미 학습한 내용을 스킬에 넣으면
**토큰을 낭비하고, 정작 중요한 지시에 대한 주의(attention)를 희석**시킨다.

| 스킬에 넣지 말 것 (모델이 이미 앎) | 스킬에 넣을 것 (모델이 알 수 없음)            |
| ---------------------------------- | --------------------------------------------- |
| 공개 API 레퍼런스                  | 사내 컨벤션, 팀 결정사항                      |
| 표준 라이브러리 사용법             | 실행 환경 제약 ("이 환경엔 X가 없다")         |
| 일반적인 디자인 패턴               | 버전별 함정 ("r128이라 CapsuleGeometry 불가") |
| 언어 문법, 잘 알려진 알고리즘      | 조직 고유 워크플로우, 출력 경로/포맷 규칙     |

**판별 테스트**: 스킬 없이 모델에게 물어봐서 이미 잘 답하면 → 그 내용은 뺀다.

---

## 원칙 2. 점진적 공개를 설계에 반영한다

- SKILL.md 본문은 **500줄 이하** 유지. 넘칠 것 같으면 계층을 하나 추가하고
  "언제 어느 파일을 읽어야 하는지" 명확한 포인터를 남긴다.
- 300줄 이상의 레퍼런스 파일에는 목차를 넣는다.
- **도메인 분기는 파일로 분리**한다. 모델이 관련 파일만 읽게 된다.

```
cloud-deploy/
├── SKILL.md              # 워크플로우 + 어떤 레퍼런스를 읽을지 선택 기준
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md          # ← Azure 작업이면 이 파일만 로드됨
```

- 결정적이고 반복적인 작업(파일 변환, 패키징 등)은 지침으로 설명하지 말고
  **scripts/에 코드로** 넣는다. 스크립트는 컨텍스트에 로드하지 않고 실행만 하면 되므로 토큰 비용이 0에 가깝다.

## 원칙 3. description이 스킬의 절반이다

트리거의 전부가 description이므로, 여기에 공을 들인다.

- **"무엇을 하는가" + "언제 쓰는가"를 모두** description에 넣는다. when-to-use 정보를 본문에 쓰면 소용없다 — 모델은 트리거 전에 본문을 읽지 않는다.
- 모델은 스킬을 **덜 쓰는(under-trigger) 경향**이 있다. 이를 보정하려면 description을 다소 "적극적으로(pushy)" 쓴다.

```yaml
# 나쁜 예 — 소극적
description: 사내 데이터 대시보드 만드는 법.

# 좋은 예 — 트리거 조건을 명시적·적극적으로 나열
description: 사내 데이터를 표시하는 빠른 대시보드 구축법.
  사용자가 대시보드, 데이터 시각화, 지표, 사내 데이터 표시를 언급하면
  '대시보드'라는 단어를 쓰지 않았더라도 반드시 이 스킬을 사용할 것.
```

- 트리거 평가를 할 거라면 **현실적인 복잡도의 프롬프트**로 테스트한다.
  단순 쿼리는 description이 완벽해도 트리거가 안 되므로 테스트 케이스로 부적합.

## 원칙 4. 명령보다 판단 기준을 준다

- 지침은 **명령형**으로 쓰되, MUST/ALWAYS를 남발하는 대신 **왜 중요한지 이유를 설명**한다.
  모델도 이유를 알면 변형 상황에서 올바르게 일반화한다.
- 특정 예시에 과적합된 좁은 규칙보다, 일반화 가능한 원칙 + 예시 조합이 낫다.
- 출력 형식은 템플릿으로, 판단이 필요한 부분은 Input/Output 예시로 보여준다.

```markdown
## 커밋 메시지 형식

**예시 1:**
Input: JWT 토큰 기반 사용자 인증 추가
Output: feat(auth): implement JWT-based authentication
```

## 원칙 5. 다른 지식 주입 수단과 역할을 나눈다

| 수단            | 적합한 지식                       | 갱신 비용           | 비고                            |
| --------------- | --------------------------------- | ------------------- | ------------------------------- |
| **스킬**        | 절차적 지식, 행동 규칙, 환경 제약 | 파일 수정 (즉시)    | 조건부 로드 → 토큰 효율적       |
| 시스템 프롬프트 | 항상 적용될 정체성·톤·전역 규칙   | 파일 수정 (즉시)    | 항상 상주 → 토큰 비용 지속      |
| RAG             | 방대한 사실 지식, 검색 대상 문서  | 인덱스 갱신         | "어떻게"보다 "무엇" 검색에 적합 |
| 파인튜닝        | 문체·형식의 깊은 내재화           | 재학습 (느림, 비쌈) | 지식 주입 용도로는 최후 수단    |

**경험칙**: 항상 필요하면 시스템 프롬프트, 상황에 따라 필요하면 스킬,
찾아봐야 하는 정보면 RAG, 행동 자체를 바꿔야 하면 파인튜닝.

## 원칙 6. 스킬도 평가한다

스킬은 쓰고 끝이 아니라 **draft → 테스트 → 평가 → 수정 반복**의 산물이다.

1. 현실적인 테스트 프롬프트 2~3개 작성 (실제 사용자가 할 법한 말)
2. **스킬 있음 vs 없음(baseline)을 나란히 실행**해 비교 — 차이가 없으면 스킬이 일을 안 하는 것
3. 객관적으로 검증 가능한 출력(파일 변환, 코드 생성)은 assertion 기반 정량 평가,
   주관적 출력(문체)은 사람 리뷰
4. **트리거 정확도는 별도로 측정** — 같은 쿼리를 여러 번(예: 3회) 실행해 트리거율을 재고,
   실패 사례를 근거로 description을 개선. train/test 분리로 과적합 방지
5. LLM 실행은 비결정적이므로 1회 결과로 판단하지 않는다 (반복 실행 분산 확인)

## 안티패턴 요약

- ❌ 모델이 이미 아는 API 문서 복붙 (→ 원칙 1 위반, 토큰 낭비)
- ❌ 모든 내용을 SKILL.md 한 파일에 (→ 원칙 2 위반, 트리거마다 전체 로드)
- ❌ when-to-use를 본문에 서술 (→ 원칙 3 위반, 트리거에 반영 안 됨)
- ❌ 소극적·모호한 description (→ under-trigger)
- ❌ MUST 도배 + 이유 없는 규칙 나열 (→ 변형 상황에서 오작동)
- ❌ 반복적 결정 작업을 자연어 지침으로 (→ scripts/로 옮길 것)
- ❌ 테스트 없이 배포 (→ baseline 비교 없이는 효과 검증 불가)
